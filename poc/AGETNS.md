技術検証(PoC) Ver1

目的:
`PixelLens` を DPI変更ツールから
画像ファイル内部構造ビューアへ発展させるため、
まず `JPEG read-only 構造解析` が成立するかを確認する

今回のPoCでは

* JPEG をバイト列として読めること
* JPEG の主要セグメントを順に解析できること
* 各セグメントの `offset` と `length` を取得できること
* 構造情報とバイト範囲の対応を持てること

を確認対象とする

---

今回の方針:

* `ImageIO` は補助用途に留める
* 構造解析の中心は `Data` 読込 + `C++ parser` とする
* まずは `read-only`
* 編集機能や再保存は対象外
* 対応形式はまず `JPEG` のみ

---

検証対象:

1.
JPEG ファイルをバイト列として読める

2.
以下の主要セグメントを走査できる

* `SOI`
* `APP0`
* `APP1`
* `DQT`
* `DHT`
* `SOF`
* `SOS`
* `EOI`

3.
各セグメントについて以下を取得できる

* `name`
* `marker`
* `offset`
* `length`
* `payload range`

4.
JPEG 全体を先頭から末尾まで破綻なく走査できる

5.
解析結果を構造一覧として出力できる

6.
解析結果から
`このノードは何バイト目から何バイト目までか`
が明確にわかる

7.
将来の UI ハイライトに使える形で
`offset-range` を保持できる

---

今回の実装範囲:

* `poc/app` に CLI ベースで実装する
* UI は作らない
* 出力はテキストまたは JSON でよい
* 最初は JPEG セグメント一覧を表示できればよい

想定コマンド例:

* `inspect-jpeg-structure <input.jpg>`
* `dump-segments <input.jpg>`

コマンド名は実装都合で変更可

---

確認項目:

1.
JPEG サンプルファイルを正常に読めること

2.
各セグメントの `offset` が実ファイル上の位置と一致すること

3.
各セグメントの `length` が JPEG 規格に沿って取得できること

4.
`SOS` 以降の圧縮データと
通常セグメントの扱いを切り分けられること

5.
`EOI` まで正しく到達できること

6.
出力結果を見れば
構造ツリー表示や hex ハイライトへ発展可能と判断できること

---

合格条件:

* `JPEG read-only parser` が動作する
* `offset / length` を安定して取得できる
* 少なくとも 1 つの JPEG で
  主要セグメントを順に列挙できる
* 解析結果を Swift 側で扱える形にできる
* Ver1 の UI で必要な `構造選択 -> バイト範囲特定`
  へ進めるだけの基礎データが取れる

---

今回やらないこと:

* DPI 編集
* EXIF 書き換え
* JFIF 更新
* PNG 対応
* TIFF 対応
* 画像保存
* バイナリ直接編集
* GUI 実装
* diff 表示

---

次段階の候補:

* `Swift <-> C++` のノードモデル橋渡し
* バイト範囲から構造ノードへの逆引き
* JSON 出力の整備
* PNG / TIFF の構造解析 PoC
